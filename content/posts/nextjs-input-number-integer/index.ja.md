{{< lead >}}
Next.jsでフォームを実装する際、`input type="number"` を使えば整数のみ入力できると思っていませんか？

実際には「e」や「--」、「.」などが入力できてしまい、意図しない値が state に入るケースがあります。

本記事では **「なぜ起きるのか」だけでなく、実務で最も安全な結論（number型を使わない選択）まで** 解説します。
{{< /lead >}}

この記事は以下のような方を想定しています。

* Next.js / Reactでフォームを実装している
* 整数入力を厳密に制御したい
* input type="number" を使って違和感を覚えたことがある

## 発生した問題

Next.jsで以下のような入力フォームを実装しました。

```tsx
<input
  type="number"
  value={value}
  onChange={(e) => setValue(Number(e.target.value))}
/>
```

要件はシンプルで、

* 整数のみ入力可能
* 小数や文字は受け付けない

というものでした。

HTML的にも意味が正しそうで、型安全そうに見えるため、
最初は迷わず `type="number"` を選びました。

しかし実際に触ってみると、

* `e`
* `--`
* `+`
* `1e10`

などが普通に入力できてしまいます。

「number型なのに、なぜ？」と混乱しました。

---

## なぜ input type="number" でも整数しか入力できないわけではないのか

結論から言うと、**HTMLのnumber型は「数値形式」を保証するだけ**で、
「整数かどうか」は保証してくれません。

### input type="number" の仕様

* ブラウザは「数値として解釈できる形式」を許可する
* 指数表記（`e`）も数値として正しい
* マイナス記号や小数点も仕様上はOK

つまり、

> 「整数のみ」という制約は、開発者が自分で実装する必要がある

ということです。

---

## よくあるが不十分な対処法

### step 属性を使う

```tsx
<input type="number" step="1" />
```

これは **UI上の操作（矢印ボタン）** には効きますが、

* キーボード入力
* コピペ

は普通に通ってしまいます。

### min / max を設定する

```tsx
<input type="number" min={0} />
```

これも **値の範囲制御** であって、整数制御ではありません。

---

## 実務で使える解消法①：type="text" + 正規表現

最も確実な方法は、**number型を使わない**ことです。

```tsx
<input
  type="text"
  inputMode="numeric"
  value={value}
  onChange={(e) => {
    const v = e.target.value
    if (/^\d*$/.test(v)) {
      setValue(v)
    }
  }}
/>
```

### ポイント

* `type="text"` にすることで `e` を根本的に防ぐ
* `inputMode="numeric"` でモバイルの数字キーボードを表示
* 正規表現で「0〜9のみ」を保証

これが **最もバグりにくい** 実装です。

### マイナスの整数も許可したい場合

```tsx
if (/^-?\d*$/.test(v)) {
  setValue(v)
}
```

---

## 解消法②：number型を使うなら onChange でガードする

どうしても `type="number"` を使いたい場合は、
**onChange 内で明示的に弾く**必要があります。

```tsx
<input
  type="number"
  value={value}
  onChange={(e) => {
    const v = e.target.value
    if (/^\d*$/.test(v)) {
      setValue(v === '' ? '' : Number(v))
    }
  }}
/>
```

ただしこの方法は、

* ブラウザ差異
* 将来の仕様変更

の影響を受けやすいため、個人的にはおすすめ度は低めです。

---

## サーバーサイドでも必ずバリデーションする

フロントで制御していても、

* API直叩き
* フォーム改ざん

は防げません。

```ts
if (!Number.isInteger(value)) {
  throw new Error('整数のみ許可されています')
}
```

**フロント + サーバーの二段構え** が前提です。

---

## 結論：整数入力を保証したいなら number 型を信用しない

* `input type="number"` は「整数専用」ではない
* 整数制御はアプリケーション側の責任
* 実務では `type="text"` + バリデーションが最も安全

Next.jsに限らず、React全般で同じ問題が起きるので、
一度ハマると忘れない知識になります。

用途別のおすすめは以下です。

* **整数を厳密に制御したい** → `type="text"` + 正規表現
* **UI的に数値入力で十分** → `type="number"` + サーバーバリデーション
* **業務システム・金額入力** → text + フォーマット制御

---

## まとめ

* number型でも `e` や `--` は仕様上入力できる
* step や min/max では不十分
* 整数限定なら text + 正規表現が最適解
* サーバー側バリデーションも必須

## FAQ

### Q1. `pattern` 属性を使えば整数入力を制限できますか？

**完全には制限できません。**

```html
<input type="number" pattern="\d*" />
```

`pattern` 属性は **type="text" のときのみ有効** です。
`type="number"` では多くのブラウザで無視されるため、
整数入力の制御には使えません。

---

### Q2. `inputMode="numeric"` を指定すると整数しか入力できなくなりますか？

**いいえ。入力制御ではなく、キーボード表示の指定です。**

`inputMode="numeric"` は主にモバイル端末で
数字キーボードを表示させるための属性です。

```tsx
<input type="text" inputMode="numeric" />
```

入力値のバリデーション自体は行われないため、
**正規表現などでの制御が必須**です。

---

### Q3. `type="number"` で `e` が入力できるのはバグですか？

**バグではなく仕様です。**

HTMLの `input type="number"` は
指数表記（例：`1e10`）を正当な数値として扱います。

そのため、

* `e`
* `+`
* `-`
* 小数点

が入力できるのは仕様通りの挙動です。

---

### Q4. `step="1"` を指定しても整数しか入力できないのはなぜですか？

`step="1"` は **UI操作（スピンボタン）にのみ影響**します。

```html
<input type="number" step="1" />
```

* キーボード入力
* コピーペースト

は制限されないため、整数入力の保証には不十分です。

---

### Q5. IME（日本語入力）使用時に注意することはありますか？

あります。

`type="text"` + 正規表現で制御している場合、
**IME変換中の値**(未確定文字列)を弾いてしまうと
入力体験が悪くなることがあります。

その場合は、

* `onCompositionStart / End` を考慮する
* 確定後にバリデーションする

といった対応が必要です。

---

### Q6. マイナスの整数も許可したい場合はどうすればいいですか？

正規表現を以下のように変更します。

```ts
/^-?\d*$/
```

* `-` を先頭1文字のみ許可
* 数字以外は拒否

用途に応じて制御を調整してください。

---

### Q7. フロントエンドで制御していればサーバー側は不要ですか？

**不要ではありません。必須です。**

フロントエンドの制御は、

* UX向上
* 入力ミス防止

が目的です。

APIの安全性を担保するため、
**サーバー側でも必ず整数チェックを行ってください。**

```ts
if (!Number.isInteger(value)) {
  throw new Error('整数のみ許可されています')
}
```

---

### Q8. 結局、実務ではどの方法を選ぶのが正解ですか？

用途別のおすすめは以下です。

* **厳密に整数のみ** → `type="text"` + 正規表現
* **簡易入力・UI重視** → `type="number"` + サーバーバリデーション
* **金額・業務系フォーム** → text + フォーマット制御

「number型＝安全」と思い込まないことが重要です。
