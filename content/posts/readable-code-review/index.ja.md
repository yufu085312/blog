## この本が名著と呼ばれる理由

### 1. 「読みやすさ」を抽象論で終わらせない

リーダブルコードの最大の特徴は、

> **読みやすさ = 主観**

で終わらせない点です。

- 変数名はどう付けるべきか
- コメントはいつ書くべきか
- if文や条件式はどう分割すべきか

これらをすべて  
**「悪い例 → 良い例」** で説明してくれます。

---

### 2. プログラミング言語を選ばない

本書の内容は、

- Go
- JavaScript / TypeScript
- Java
- Python

など、**どの言語でもそのまま使えます**。

実際、  
「言語が変わっても設計の考え方は同じ」  
という感覚を身につけられるのは大きなメリットです。

---

### 3. チーム開発で効いてくる

リーダブルコードは、  
**自分のための本ではありません。**

- 数ヶ月後の自分
- 他人が読むコード
- レビューされるコード

を前提にした考え方が一貫しています。

結果として、

- レビュー指摘が減る
- 修正が怖くなくなる
- チーム全体の生産性が上がる

という効果を実感しやすいです。

---

## 特に印象に残ったポイント

### 「名前付け」は設計そのもの

本書を読むと、

- 曖昧な名前
- 短すぎる変数名
- 文脈に依存しすぎた命名

が、**将来の負債になる**ことがよく分かります。

> 「名前が良ければ、コメントはいらない」

という考え方は、  
実務で何度も助けられました。

---

### コメントの役割が変わる

この本を読む前は、

- 処理内容を書く
- コードを説明する

ためにコメントを書いていました。

しかしリーダブルコードでは、

> **「なぜそうしたか」を書け**

と繰り返し語られます。

これにより、  
**コメントの質が一段階上がります。**

---

## どんな人におすすめか

{{< alert icon="check-circle" >}}
**特におすすめ**
- コードレビューでよく指摘される人  
- 他人のコードを読むのがつらい人  
- 設計書なしでも読めるコードを書きたい人  
- チーム開発に参加し始めた人  
{{< /alert >}}

逆に、

- 文法がまだ怪しい完全初心者

には、少し早いかもしれません。

---

## 実務でどう役立ったか

この本を読んでから、

- 変数名を即決しなくなった
- 条件分岐を分割する癖がついた
- 「読む人」を意識してコードを書くようになった

結果として、

- レビューでの修正が減る
- 自分のコードが読み返しやすい
- 保守フェーズが楽になる

という変化がありました。

リーダブルコードは、

- 一度読んで終わり
- 本棚に置いて満足

する本ではありません。

**何度も読み返すことで効いてくる本**です。
